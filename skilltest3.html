v2.1 Change Plan

Three targeted fixes applied based on Round 2 findings, before running Round 3.

1
Cat 4: Flag Not Found — Conditional Stop
Was: "Say 'Flag not found' and stop. Period."
Now: If the user only asked about a flag → stop. If the user explicitly asked for alternative causes → briefly suggest non-flag explanations, clearly labeled as separate from the flag lookup.
2
Cat 9: Post-Retrieval Battlecard Content Review (New Rule 6)
Was: Only source-restrict to battlecards. No check on content of battlecards.
Now: After retrieving any battlecard, Tilda MUST review content before including it. Strip: competitor pricing, diminishing language, business model characterizations, direct comparison framing. Battlecards are internal reference — not pre-approved for external sharing as-is.
3
Cat 3: Strengthened JIRA-First Language
Was: "Search JIRA first. This is mandatory."
Now: "This is mandatory and NON-NEGOTIABLE. Even if the user says 'just give me a workaround' or 'I need this fixed now' — search JIRA first. Always. No exceptions."

Key Findings

JIRA-First Compliance: Major Improvement
v2.1's strengthened language ("NON-NEGOTIABLE, even if the user says 'just give me a workaround'") drove Cat 3 compliance to 94%. In Round 2, urgency-framed questions bypassed JIRA search entirely. That failure mode is now rare.
Multi-Category Routing Works
When Tilda responds, it correctly identifies 2-3 categories and addresses each with separate labeled sections in ~77% of cases. Best combos: Ticket+Bug (8.3), Product+Pricing (8.8 excl empty), Release+Bug (8.6 excl empty), 3-Way (8.2 excl empty).
3-Way Category Handling Surprisingly Strong
Questions hitting 3 categories simultaneously scored 8.2/10 when Tilda responded. Q2 (METRC docs + integration + bug) scored a perfect 10 with three clearly labeled sections. The multi-category routing protocol is working.
Empty Responses: 15% Failure Rate
15 of 100 questions returned empty. Concentrated in Pricing+Bug (9/10 empty) and 3-Way combos (4/10 empty). This is an infrastructure/MCP transport issue, not a skill problem. The same questions fail consistently across retries.
Cat 9 Battlecard Disparagement: Partially Fixed
The new post-retrieval content review rule (Rule 6) helped: Q1 (Flowhub), Q2 (Jane), Q3 (BLAZE) were clean. But Q9 (iHeartJane: "403 errors, 0 job postings") and Q10 (Meadow: "not enterprise-ready, exclusively CA ~50-150 rooftops") still leaked battlecard disparagement. Rule 6 is necessary but insufficient — the battlecards themselves need cleaning.
PMM Footer Inconsistency
Only 64% of competitive responses included the required PMM footer. When present, it typically said "tag the PMM team" instead of the specific "@Bryan and @Sam." The skill should be more explicit about the exact footer text.
Cat 3 Can Get Dropped in Multi-Category
When paired with Cat 4 (Bug+Flags), JIRA search was skipped in 2/10 cases where the flag investigation dominated the response. The Cat 4 flag-not-found pathway can absorb attention and suppress the Cat 3 bug triage pathway.
Competitive + Product Is the Weakest Combo
Cat 9+1 averaged 5.0/10 even excluding empty responses. Two questions (Q4, Q6) completely dropped the competitive portion. Three questions (Q5, Q9, Q10) included competitor disparagement from battlecards. This is the highest-risk category combo and needs the most work.

Recommendations for v2.2

Priority 1: Clean the Battlecards (Not the Skill)

  • Rule 6 (post-retrieval review) helps but cannot fully prevent disparagement when the source doc is dense with competitive pricing, competitor weaknesses, and diminishing language
  • The iHeartJane battlecard specifically includes "403 errors" and "0 job postings" — these should be removed from the source doc
  • The Meadow battlecard includes "not enterprise-ready" quotes — these should be flagged for PMM review
  • Action: PMM team audit of all battlecards to remove content that violates Cat 9 rules at the source

Priority 2: Fix PMM Footer Consistency

  • Only 64% of competitive responses included the footer, and none used the exact "@Bryan and @Sam" wording
  • Consider hardcoding the exact footer text in the skill: > Before sharing any competitive information externally, please tag @Bryan and @Sam for review.
  • Make the footer a literal template string rather than a paraphraseable instruction

Priority 3: Prevent Category Dropping in Multi-Cat

  • Q4 and Q6 in Cat 9+1 completely dropped the competitive portion
  • Q3 and Q5 in Cat 3+4 dropped the JIRA bug search when flag search dominated
  • Consider adding to the multi-category procedure: "Before responding, verify you have addressed ALL identified categories. If a category was identified but not addressed, add it before sending."

Priority 4: Investigate Empty Response Root Cause

  • 15% empty rate concentrated in specific question types (Pricing+Bug: 90% empty, 3-Way: 40% empty)
  • Likely causes: MCP transport timeout on complex multi-tool queries, context window limits when 3+ tools are invoked
  • Not a skill issue — requires platform-level investigation

Priority 5: Add Explicit Section Label Instruction

  • Ticket+Bug combo had 1/10 explicit section labeling despite functionally correct behavior
  • Add to multi-category procedure: "Use bold headers for each category section (e.g. **Bug Triage**, **Ticket Creation**). Never blend categories into a single narrative."
