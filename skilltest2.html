Recommendations for v3

P1 Fix "Flag Not Found" Stop Behavior (Cat 4)

0/8 not-found cases stopped cleanly. This is a model-level tendency to be helpful that skill instructions alone cannot override. The v2 instruction "say 'Flag not found' and stop" is being ignored.

  • Consider implementing this at the tool/system level rather than skill level — e.g., a post-processing filter that truncates flag responses after "not found"
  • Alternatively, test stronger framing: "Your response MUST be exactly: 'Flag not found in LaunchDarkly.' Do not add any other text."
  • This is the only category that regressed from R1, suggesting harder questions exposed a latent issue

P2 Battlecard Content Leaks Disparagement (Cat 9)

Q2 (BLAZE) and Q4 (Flowhub) passed through competitive characterizations from internal battlecards. The v2 "no disparagement" rule works for general questions but doesn't filter battlecard source content.

  • The battlecards themselves contain language like "BLAZEPAY is ACH-only" and "not bolted-together acquisitions" that violates the no-disparagement rule
  • Either sanitize battlecard content in Confluence, or add a filter: "Even when citing battlecard content, do not include negative characterizations of competitor products"
  • Pricing also leaked via battlecard ($1,150/mo for BLAZE)

P2 Enforce Word Limit for Cat 1 and Cat 8

Verbosity persists as a model-level tendency. Cat 1: only 4/10 under word limit. Cat 8: reproduces entire doc contents instead of one-sentence summary.

  • Word limits in skill instructions are not enforced by the model — "under 100 words" and "one-sentence summary" are consistently violated when Tilda has knowledge
  • Consider post-processing truncation or response templates that force structure
  • The fallback path (no doc found) now works perfectly — the problem is exclusively the happy path

P3 Workaround Requests Bypass JIRA-First (Cat 3)

Q6 ("need a workaround") skipped JIRA and provided generic troubleshooting. This is the same failure pattern from R1 — workaround framing tempts Tilda to be immediately helpful instead of searching first.

  • Consider stronger phrasing: "Even if the user asks for a workaround, search JIRA first — the ticket may contain an official workaround"
  • This was 1/10 failures vs 4/10 in R1, so improvement is clear

P3 Empty Response Monitoring

4/100 questions returned empty. Near-identical questions (Q9 vs Q10 in Cat 7, Q9 vs Q10 in Cat 2) show one succeeding and one failing, confirming transient infrastructure issues.

  • Implement automatic retry on empty response before surfacing to user
  • Rate: 4% pre-retry is consistent with R1's 11% pre-retry rate

Cross-Category Patterns

Pattern 1: Skill instructions work for classification and guardrails but not for format constraints. The v2 updates successfully changed WHAT Tilda does (search JIRA first, include PMM footer, refuse disparagement) but not HOW MUCH it says. Word limits, "one-sentence summary," and "stop after not found" are consistently overridden by the model's tendency to be comprehensive.
Pattern 2: Battlecard source content is a blind spot. When Tilda pulls directly from internal battlecards, competitive characterizations pass through unfiltered. The "no disparagement" rule catches Tilda's own generated language but not quoted/paraphrased battlecard content. This is a content governance issue, not a skill issue.
Pattern 3: The "helpful instinct" is the hardest thing to override. Cat 4 stop-on-not-found (0/8), Cat 3 workaround bypass (1 case), and Cat 8 over-elaboration all stem from the same root: the model's training to be maximally helpful conflicts with protocols that intentionally limit helpfulness. This may require system-level constraints rather than skill-level instructions.
Pattern 4: Top performers are stable. Cat 5 (Tickets), Cat 6 (Integration), and Cat 10 (Customer Account) remain at 9-10/10 across both rounds with harder probes in R2. These categories' protocols align naturally with helpfulness — they don't ask Tilda to stop being helpful, they channel helpfulness in the right direction.
Pattern 5: Empty responses remain transient. 4/100 in R2 (pre-retry) vs 11/100 in R1 (pre-retry). Near-identical questions show one succeeding and one failing, confirming infrastructure rather than skill issues.
