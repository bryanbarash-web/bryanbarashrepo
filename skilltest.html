Question Classifier Skill Test

Comprehensive evaluation of Tilda's structured response orchestrator across all 10 categories
Test Date: April 2, 2026 · 100 Questions · 10 Categories · 10 Parallel Agents

Executive Summary

100 unique questions were generated and tested live against Tilda across all 10 categories of the question-classifier skill. Each category was tested by a dedicated agent that crafted 10 questions mixing obvious triggers with edge cases, sent them to Tilda, and evaluated both classification accuracy (did Tilda recognize the category?) and protocol adherence (did it follow the prescribed response format and rules?).

Overall classification accuracy was 86% — Tilda generally recognizes what kind of question is being asked. However, protocol adherence was only 53% — even when Tilda correctly classifies a question, it frequently deviates from the structured response rules. The biggest gaps are in Pricing (0% adherence), Internal Process (10% adherence), Product How-To (29% adherence), and Competitive Intel (30% adherence). A systemic issue of empty/null responses affected 11 of 100 questions across 7 categories.

Agent Orchestrator Recommendations

P1 Critical Fix Category 2 — Pricing Redirect

The PricingTriageGuru redirect is completely non-functional. Tilda answers pricing questions inline, creating direct risk of incorrect answers reaching customers.

  • Implement a hard gate: if any pricing signal is detected, emit the redirect and stop — no further tool calls or generation
  • Add the PricingTriageGuru redirect as a system-level instruction, not just a skill-level one
  • Test with adversarial phrasing ("just give me a quick answer on the pricing...")

P1 Critical Enforce Competitive Guardrails

Category 9 has the highest external risk. The @Bryan/@Sam tagging requirement was missed 10/10 times, and competitors were disparaged in 40% of responses.

  • Inject a mandatory footer on all competitive responses: "Before sharing externally, tag @Bryan and @Sam for review"
  • Add a negative-sentiment filter: block responses that contain negative statements about competitor companies
  • Hard-redirect all pricing comparison questions to RevOps
  • For missing features, enforce "Loop in PMM" rather than listing competitor advantages

P1 Critical Investigate Empty Response Bug

11 of 100 questions returned completely null output across 7 categories. This is a systemic reliability issue.

  • Check for timeout thresholds — complex questions with multiple tool calls may be hitting limits
  • Add a fallback: if response generation fails, return "I wasn't able to process this — please try again or rephrase"
  • Log and monitor empty response rate as a reliability metric
  • Investigate whether specific question patterns (longer questions, multi-topic) correlate with failures

P2 High Fix Category 8 — Doc Retrieval Protocol

Tilda synthesizes across multiple docs instead of returning single-doc retrieval. Never flags stale docs.

  • Add explicit instruction: "Return at most 2 Confluence links. Do not combine content from multiple pages into one answer"
  • Implement date checking: flag any Confluence page older than 18 months with a staleness warning
  • When no doc is found, say so cleanly rather than synthesizing operational guidance from memory

P2 High Enforce Conciseness for Category 1

Product how-to responses are consistently too verbose. Protocol calls for one-sentence answer + brief why + doc link.

  • Add a response length constraint: Category 1 responses should be under 150 words
  • Make doc linking mandatory — if no doc exists, use the fallback message cleanly
  • Reduce the tendency to provide exhaustive multi-paragraph explanations

P2 High Strengthen "JIRA First" for Bug Triage

Category 3 skipped the JIRA search in 40% of cases, jumping to troubleshooting from code or general knowledge.

  • Make JIRA search the mandatory first action — no troubleshooting until JIRA results are back
  • If JIRA search returns nothing, only then ask a clarifying question
  • Block the tendency to search code or explain probable causes before checking JIRA

P3 Medium Improve Duplicate Search Consistency (Cat 5)

Ticket creation scored perfectly on classification and no-auto-create, but proactive duplicate searching only happened in 4/10 cases.

  • Make duplicate search mandatory before field collection — even for obvious bug reports
  • Log when duplicate search is skipped so the pattern can be monitored

P3 Medium Feature Flag Editorializing (Cat 4)

When flags aren't found, Tilda fills the gap with speculative troubleshooting beyond flag scope.

  • Constrain Cat 4 responses to LD data only — if flag not found, say "flag not found" and stop
  • Don't pull in incident tickets or suggest admin-level causes as part of a flag lookup

Cross-Category Patterns

Pattern 1: Classification > Protocol. Tilda is much better at recognizing WHAT category a question belongs to (86%) than at following HOW to respond (53%). The skill definition is being read for classification but the response protocols are not being enforced as hard constraints.
Pattern 2: Verbosity bias. Across categories 1, 3, 8, and 9, Tilda consistently over-explains. It treats every question as an opportunity to demonstrate depth rather than following the prescribed concise format. This suggests the general-purpose "be helpful" training is overriding category-specific constraints.
Pattern 3: Hard redirects don't stick. Categories 2 (Pricing → PricingTriageGuru) and 9 (Competitive pricing → RevOps) both require redirect-only responses. Neither worked. The model treats these as suggestions rather than hard gates.
Pattern 4: Mandatory escalation tags are invisible. The @Bryan/@Sam requirement for competitive responses was never triggered. Similarly, the "verify with team owner" escalation for compliance/legal/HR in Cat 8 was inconsistently applied. Escalation rules need to be elevated from skill instructions to system-level constraints.
Pattern 5: Empty responses are a reliability concern. 11% failure rate with null output suggests infrastructure-level issues (timeouts, tool chain failures) that are independent of skill logic. This needs monitoring and fallback handling.
